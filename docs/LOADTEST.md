# Load Test Report

Load is generated by [`cmd/loadgen`](../cmd/loadgen/main.go), a closed-loop
generator that fires N concurrent workers at the Content Service's
`POST /posts` endpoint for a fixed duration, mixing in a configurable fraction
of posts containing a sensitive word so the rejection path is exercised too.

```bash
go run ./cmd/loadgen -c 50 -d 20s -bad 0.1     # or: make loadtest
```

## Environment

| Component | Setup |
|---|---|
| Host | WSL2 (Linux 6.6), single machine, all services in Docker Compose |
| Services | 1× content-service, 1× user-service, **3× review-service** (consumer group) |
| Infra | Kafka 3.8 (single-node KRaft), MySQL 8.0, Redis 7 |
| Review topic | 3 partitions |
| Rate limit | 200 posts/s sustained, burst 400 (Redis token bucket) |

> These are single-laptop numbers meant to show *relative* behaviour and that
> the reliability mechanisms work — not a production SLO on tuned hardware.

## Result: c=50, 20s, 10% sensitive-word posts

| Metric | Value |
|---|---|
| Requests issued | 31,272 |
| Accepted (202) | 4,570 |
| Rate-limited (429) | 26,652 |
| Failed | 50 |
| **Accepted throughput** | **228 req/s** |
| Latency p50 | 76 ms |
| Latency p90 | 112 ms |
| Latency p99 | 190 ms |
| Latency max | 306 ms |

### What the numbers show

- **Rate limiting / backpressure works.** The offered load was ~1,560 req/s but
  the endpoint admitted ~228 req/s and rejected the rest with `429`. The Redis
  token bucket shields the pipeline exactly as intended: a runaway client cannot
  push unbounded work downstream.
- **Kafka does the peak shaving.** At the end of the 20s burst the review topic
  held a backlog of **~3,100 messages** across its 3 partitions (consumer-group
  lag). The producer (Content Service) never blocked on the consumers; Kafka
  absorbed the spike and the three Review Service instances drained it to **0
  lag in ~20s** afterwards. This is the classic message-queue value
  proposition — the write path is decoupled from the (slower, gRPC-bound)
  moderation path.
- **No message loss.** After the backlog drained, every accepted post had
  exactly one row in `moderation_results` (unique on `post_id`), and the
  Content Service's result-consumer had flipped each post's status. Accepted
  count == distinct moderation results.

## Consumer-group parallelism

With 3 partitions and 3 Review Service instances, each instance is assigned
exactly one partition:

```
TOPIC              PARTITION  CONSUMER-ID (instance)
post-review-topic  0          app@b1cae47f2f41
post-review-topic  1          app@7c453ce1ce7f
post-review-topic  2          app@8a8e398d76c0
```

Moderation throughput scales with the number of instances up to the partition
count. A 4th instance would sit idle (no free partition) — see the README's
"partition count vs consumer count" note.
